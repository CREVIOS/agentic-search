## Agent runs (real corpus, real LLM, grounded citations)

### Claude Agent SDK over MCP stdio (`examples/claude_agent_corpus.py`)

**Prompt** — *"Across the corpus, find ONE concrete example each of how
the Rust async ecosystem, the Web Platform (MDN), and the Kubernetes API
model the same idea of bounded queues with backpressure. Cite an exact
filename and a short quoted passage."*

**13 MCP tool calls** (every one against `s3://agentic-search-it/corpus`):
`grep` × 8, `ls` × 2, `read` × 3.

**Final answer (excerpt):**

> **Rust async (Tokio)** — `corpus/tokio-tutorial/channels.md`
> > *"The channel is created with a capacity of 32. If messages are
> > sent faster than they are received, the channel will store them."*
> > Section heading line 430: *"Backpressure and bounded channels"*.
> > `mpsc::channel(32)` blocks senders when buffer full = backpressure.
>
> **Web Platform (MDN Streams)** —
> `corpus/mdn-webapi/streams_api/concepts/index.md`
> > *"An important concept in streams is **backpressure** — this is the
> > process by which a single stream or a pipe chain regulates the
> > speed of reading/writing."*
> > Bound mechanism = `highWaterMark` queuing strategy.
>
> **Kubernetes** —
> `corpus/k8s-concepts/cluster-administration/flow-control.md`
> > *"`kube-apiserver` flags `--max-requests-inflight` and
> > `--max-mutating-requests-inflight`. With APF enabled, the
> > concurrency limits cap concurrent requests; excess rejected with
> > 429 or queued by API Priority and Fairness."*
>
> **Synthesis.** All three cap an in-flight buffer with explicit
> numeric limit (channel capacity / highWaterMark / max-inflight) so
> producers cannot outrun consumers. When limit hit, system pushes
> signal upstream — sender awaits, stream reports `desiredSize<=0`,
> apiserver returns 429 — converting unbounded queueing into flow
> control.

### DeepAgents over REST (`examples/deepagents_corpus.py`)

**Prompt** — *"Find every place in the corpus that discusses
backpressure or rate limiting between async producers and consumers.
Cite 3 distinct files. Synthesise the trade-off."*

**Cited files** (all verified to exist on S3):

| # | File | Size on S3 |
|---|---|---:|
| 1 | `corpus/mdn-webapi/bytelengthqueuingstrategy/bytelengthqueuingstrategy/index.md` | 1,831 B |
| 2 | `corpus/mdn-webapi/countqueuingstrategy/countqueuingstrategy/index.md` | 1,297 B |
| 3 | `corpus/k8s-concepts/cluster-administration/flow-control.md` | 42,566 B |

**Final answer (excerpt):**

> **Synthesis.** All three sources warn that if you let a fast
> producer enqueue work without bound, you risk memory exhaustion or
> server crash — the system must impose a ceiling (a `highWaterMark`
> or an inflight-request limit) to protect the consumer. The
> trade-off is that setting the ceiling too low throttles throughput
> and can cause spurious rejections during legitimate bursts, while
> setting it too high (or omitting it) defeats the protection
> entirely.

✅ Real cite. Real quotes. Real S3 protocol on the wire (every grep /
read call signed SigV4 against RustFS at `:19000`).

## Known limit hit (and the honest fix)

`/search` (planner with three parallel grep probes) is borderline on a
10k-file corpus with no centroid vector index built. The first cold run
of the natural-language query timed out at the 15 s endpoint budget
the server enforces (the right behaviour — fail fast rather than hold
the agent open). Warm subsequent runs return inside the budget because
the OS page cache + manifest are hot.

The principled fix at this scale is the vector path (`agentic-search
index s3://…/corpus`), which trades grep-time-per-query for a one-time
build that gives constant-time recall. The bench bin
(`bench/src/bin/sift1m.rs`) shows the centroid index lands at 97.25 %
recall@10 / 4.85 ms p50 warm on 1M × 128d vectors; the same path runs
on this 10k-doc corpus once we build it (skipped in this run to keep
total wall-clock under 10 minutes).
